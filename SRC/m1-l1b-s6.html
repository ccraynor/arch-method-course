Quick Reference: Output Style Gate Conventions

1. Artifact Naming: MRHN_[ArtifactName]_v[Version]_[Date]. Four segments. CamelCase name, no spaces. Integer version starting at 1. Date format: YYYY-MM-DD.

2. SI Identifiers: SI-01 through SI-16. Assigned in Bucket Map order. Never reused; retired identifiers leave a gap in the sequence.

3. DC Identifiers: DC-01 through DC-48. Assigned during Module 2 decomposition, grouped by SI parent. Same retirement rule as SI identifiers.

4. Governance Record: Every decision requires all five fields: Decision Statement, Constraint Reference, Rationale, Risk Note, Governance Checkpoint Linkage.

5. Version Control: Revisions happen only after documented governance review. Create a new file with an incremented version. Never edit a previous version in place.

Snapshot Review: Task 3

Expert Governance Decision Record: MRHN-D-001

MRHN-D-001: IR-2024 Version Alignment
1 Decision Statement
The MRHN course aligns to IR-2024 as adopted in January 2024. Revisions to IR-2024 occurring after project start are out of scope without a documented scope change request and governance review.
2 Constraint Reference
RC-2 (IR-2024 as adopted); PC-1 (IR-2024 protocol alignment); LC-1 (4 hours total SME access)
3 Rationale
The OCR remediation timeline (RC-3) and SME access cap (LC-1) mean scope cannot flex after the first SME meeting. Aligning to IR-2024 as a fixed document prevents drift that cannot be absorbed within project constraints. The client brief names IR-2024 by adoption date, establishing it as the governing reference.
4 Risk Note
If Meridian updates IR-2024 during the project window, the delivered course will not reflect the change. This risk is accepted given the timeline and SME constraints. Any post-launch protocol update requires a documented v2 revision under the version control protocol.
5 Checkpoint Linkage
Gate 1.TG.1 (Transition Gate, Module 1) will verify that all performance items trace to IR-2024 as adopted. Module 3 Outcome Mapping will confirm no performance item references a protocol version not covered by the course content.

Your Development

What your response reveals: your reasoning shows whether you can tie a decision to a specific constraint and name its risk before a reviewer does. What to carry forward: a decision record with no risk note is a note, not governance.

Compare a weak record to a strong one

Before
  • Decision Statement: The course will use the IR-2024 protocol.
  • Constraint Reference: The client wants the current incident response protocol.
  • Rationale: IR-2024 is the protocol Meridian uses, so the course should match it.
  • Risk Note: If something changes later, the course might need to be updated.
  • Checkpoint Linkage: This will be reviewed before the course is finished.

This record names no constraint code, restates the decision as its rationale, and gives a risk note with no concrete failure condition.

After
  • Decision Statement: The MRHN course aligns to IR-2024 as adopted in January 2024. Protocol revisions after project start are out of scope without a documented scope change request and governance review.
  • Constraint Reference: RC-2 (IR-2024 as adopted); PC-1 (IR-2024 protocol alignment); LC-1 (4 hours total SME access).
  • Rationale: The OCR remediation timeline and the SME access cap mean scope cannot flex after the first SME meeting. Fixing IR-2024 as of its adoption date prevents drift the project cannot absorb. The brief names IR-2024 by adoption date, which makes it the governing reference.
  • Risk Note: If Meridian revises IR-2024 mid-project and the change is absorbed silently, learners will train on a protocol version that does not match the audited standard. That creates a compliance gap during the OCR review. The risk is accepted; any post-launch update requires a documented v2 revision.
  • Checkpoint Linkage: Gate 1.TG.1 verifies that every performance item traces to IR-2024 as adopted. Module 3 Outcome Mapping confirms no performance item references a protocol version the course content does not cover.

This record cites the binding constraints by code and names a concrete failure condition a reviewer can check.

Why it is stronger

The strong version cites the exact binding constraints by code and names a concrete failure condition, learners trained on an unaudited protocol version during the OCR review, so a reviewer can both verify the decision and see precisely what the risk note protects against.
