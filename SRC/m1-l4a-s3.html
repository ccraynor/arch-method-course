Decompose Bucket 1: How Calibration Annotations Govern Every DC Component Decision

Phase 2, Step 6, Part 1: Bucket 1 (DC-01 through DC-15)
Scenario: Watch an expert work at Meridian Regional Health Network. MRHN_PerformanceArc_v1 sets the ceiling. The expert decomposes the five Bucket 1 scope items (SI-08, SI-01, SI-16, SI-06, SI-02) into DC-01 through DC-15. You see the full decomposition methodology for SI-08 and SI-06. These are the two Analysis-level items with calibration annotations from MRHN-D-003. The remaining items appear in summary.
Show Expert Reasoning

The calibration note for a scope item reads like background detail. So why does the expert treat it as a hard constraint on every component?

01

What Step 6 Does, and the Rule That Governs Every Component Decision

Step 6 converts scope items into discrete, teachable units. Each DC component represents one learnable capability, specific enough that Module 2 can design instruction and assessment for it independently, bounded enough that it fits within a coherent lesson without requiring all surrounding components to be present.

The governing rule is this: the calibration annotation for a scope item is not context. It is a constraint. If MRHN-D-003 assigned SI-08 at Analysis level because the MRHN population lacks practiced pattern recognition, then every DC component derived from SI-08 must require analytical practice, not procedure-following. A component that could be satisfied by consulting a reference table has not honored the taxonomy assignment. The calibration governs both what each component requires and what it excludes.

The arc from Step 5 sets a second constraint. Each component must be traceable to the Bucket 1 sub-arc: "Evaluate indicators of a potential cybersecurity incident to distinguish it from non-incident system events, apply the MRHN severity rubric to produce a documented severity classification, and evaluate whether Tier 1 containment capacity is sufficient." Components that do not build toward this arc should not exist. Components whose absence creates a gap in the arc are missing and must be added.

02

Deep Dive: Decomposing SI-08 (Threat Identification)

SI-08 Threat Identification Analysis
MRHN-D-003 Flag Healthcare IT professionals at MRHN must distinguish a cybersecurity incident from system failures, user errors, and hardware problems without a practiced pattern library. Analysis level is assigned because identification requires active evaluation of competing event hypotheses, not pattern matching from experience. Instruction and assessment must target the evaluation process, not the conclusion.
Arc Constraint Bucket 1 sub-arc requires "evaluate indicators... to distinguish it from non-incident system events." Components must build this evaluation capability, not recognition speed.
Expert Reasoning: How Many Components?

SI-08 at Analysis level breaks into four components because the full identification capability has four distinct learnable parts that can fail independently. A learner can know the MRHN event taxonomy (DC-02) but fail at ambiguous-indicator situations (DC-03). A learner can reach a correct identification but produce a documentation rationale that does not name the competing hypotheses they ruled out (DC-04), which fails the RC-1 documentation requirement and leaves the response team without the identification record they need.

I do not produce a fifth component for "confirm identification with a supervisor." That capability is excluded by PC-3 authority boundaries: Tier 1 responders at MRHN are expected to reach identification independently. Adding a confirmation component would teach a dependency that the constraint set explicitly does not permit in a Tier 1 response.

Calibration Constraint Applied: What Each Component Must Require

The MRHN-D-003 annotation governs all four components: none can be satisfied by applying a decision tree or lookup table without analytical judgment. DC-01 must require distinguishing among event types, not just identifying one. DC-02 must require applying the taxonomy to assign a type, not just recognizing a named category. DC-03 is the highest-demand component: it is entirely about what a learner does when evidence points in two directions simultaneously, the situation the calibration annotation identified as the critical gap for this population. DC-04 must require that the documentation names the alternatives that were evaluated, not just the conclusion reached.

DC-01 Analysis

Recognize the indicator profile of a cybersecurity incident by evaluating whether event characteristics match a security event pattern, a system failure pattern, a user error pattern, or a hardware fault pattern using the MRHN event taxonomy as the evaluation framework.

Calibration: must require active comparison among event types, not single-category identification.
DC-02 Application

Apply the MRHN threat classification schema to assign an incident type designation (unauthorized access, malware infection, ransomware, data exfiltration, denial of service, or insider threat) based on evaluated indicator evidence.

DC-03 Analysis

Evaluate competing event hypotheses when initial indicator data is ambiguous or consistent with more than one event type, using the MRHN hypothesis-evaluation protocol to determine which explanation best accounts for the full evidence set.

Calibration: this component directly addresses the critical gap identified in MRHN-D-003. Assessment must present ambiguous indicator sets, not clean single-hypothesis scenarios.
DC-04 Analysis

Document the threat identification determination with criteria-based rationale that names the event type assigned, the indicators evaluated, the competing hypotheses considered and ruled out, and the evidence basis for the conclusion, satisfying RC-1 documentation requirements.

Calibration: documentation must name ruled-out hypotheses, not just state a conclusion. A record that only states "identified as ransomware" is insufficient.

SI-08 Decomposition Complete: 4 components produced

SI-08 decomposed into four Analysis-level components because the MRHN population must distinguish genuine indicators from noise, not just follow a procedure.

03

Deep Dive: Decomposing SI-06 (Escalation Decision)

SI-06 Escalation Decision Analysis
MRHN-D-003 Flag The Tier 1 / Tier 2 escalation decision at MRHN is Analysis level because it requires evaluating whether the current team's capability is sufficient for the confirmed incident, not simply applying an escalation checklist. The healthcare IT professional must assess team readiness against incident complexity in real time, without the benefit of a pre-determined threshold that resolves the judgment.
Arc Constraint Bucket 1 sub-arc: "evaluate whether Tier 1 containment capacity is sufficient, escalating to Tier 2 when indicated and documenting the escalation decision with criteria-based rationale." Components must build evaluation of capacity, not criteria-lookup.
Expert Reasoning: Four Components, Not Two

A naive decomposition of SI-06 produces two components: "apply escalation criteria" and "document the decision." That decomposition misses the calibration constraint entirely. The MRHN-D-003 annotation does not say escalation criteria don't exist. It says the decision cannot be resolved by criteria alone, because the critical variable is whether the current team can execute a Tier 1 response to this specific incident.

That means there must be a component for assessing team capability (DC-10) and a separate component for evaluating whether incident complexity and scope exceed that capability (DC-11) before a learner reaches criteria application (DC-12). These are not the same cognitive act, and they can fail independently. A learner can accurately assess their team's capability and still misapply the criteria. A learner can correctly apply criteria and still produce an escalation rationale that omits the capability evaluation, failing the RC-1 requirement and leaving the Tier 2 team without the context they need to take over.

DC-13 (documentation) is not a simple record-keeping task. The arc specifies "criteria-based rationale": the documentation must name the capability factors evaluated, the criteria applied, and the conclusion reached. This is Evaluation-adjacent for the documentation component specifically, but I hold it at Analysis because the judgment is made in DC-10 through DC-12; DC-13 requires translating that judgment into a governed record, which is Application-level execution of an Analysis-level decision.

Notice this: decomposing from the plain label gives you two components, but the calibrated item needs four.
DC-10 Analysis

Assess current Tier 1 response team capability against the demands of the confirmed incident by evaluating available personnel, technical access, authority scope under PC-3, and relevant experience, producing a team-readiness determination for this specific incident.

Calibration: this is a real-time self-assessment, not a staffing checklist. The evaluative object is team readiness against a specific incident, not team composition in the abstract.
DC-11 Analysis

Evaluate whether the incident's confirmed complexity, scope, potential legal exposure (HIPAA, IR-2024), and operational impact exceed Tier 1 authority and capacity as assessed in DC-10, identifying the specific factors that drive escalation consideration.

Calibration: evaluation must connect to the team-readiness determination from DC-10, not just assess incident severity in isolation.
DC-12 Application

Apply the MRHN escalation criteria to the team-readiness and incident-complexity evaluations to determine whether Tier 1 response continues or Tier 2 escalation is required, producing a clear escalation status.

DC-13 Application

Document the escalation decision with criteria-based rationale that names the team capability factors evaluated, the incident complexity factors evaluated, the MRHN criteria applied, and the escalation status reached, satisfying RC-1 requirements and providing the Tier 2 team with the handoff record they require.

Arc constraint: rationale must name capability factors, not just state the conclusion. "Escalating to Tier 2" without reasoning is insufficient per arc specification.

SI-06 Decomposition Complete: 4 components produced

SI-06 decomposed into four components that separate recognizing the escalation trigger from executing and documenting the handoff.

04

Remaining Bucket 1 Items: SI-01, SI-16, SI-02

The three remaining items carry Application-level assignments and no MRHN-D-003 calibration flags. Each decomposes more directly, because the cognitive demand is procedure execution within a known framework, not evaluation under ambiguity. I show the component logic and note where the arc or constraint set still shapes the decomposition.

SI-01 Initial Incident Detection Application

Two components. SI-01 is an Application-level procedural item: learners must execute the MRHN detection procedure correctly and initiate the incident record that all downstream documentation depends on. The two components fail independently. A learner can execute the detection checklist and fail to activate the log per RC-1, which breaks the documentation chain.

DC-05 Application

Execute the MRHN initial detection procedure for a confirmed or suspected cybersecurity event, verifying all detection checklist items are addressed and no detection criteria are skipped before proceeding to identification.

DC-06 Application

Activate the MRHN incident log and assign a preliminary incident tracking ID per RC-1 documentation requirements, establishing the governed record that all subsequent triage and response documentation will reference.

SI-16 Severity Assessment Analysis

Three components. SI-16 is assigned at Analysis level because the MRHN severity rubric requires evaluating contextual factors, active patient care status, facility type, system criticality, that can modify the rubric output. A learner who only applies the rubric criteria without evaluating context may assign an incorrect severity level when, for example, a moderate-severity indicator profile occurs at a facility with active patient care on affected systems. DC-08 is the component that captures this contextual judgment.

DC-07 Application

Apply the MRHN severity rubric criteria to the confirmed incident to produce a preliminary severity level (Critical, High, Moderate, or Low) based on the four rubric dimensions: scope, affected system type, data exposure potential, and operational impact.

DC-08 Analysis

Evaluate contextual factors, active patient care status at the affected facility, clinical system interdependencies, and multi-site operational exposure, to determine whether any factor warrants adjusting the preliminary rubric severity level.

Analysis level: contextual evaluation can override the rubric output. Instruction must present scenarios where rubric and context point in different directions.
DC-09 Application

Document the confirmed severity classification with rubric criteria notation, contextual factor record, and any adjustment rationale, satisfying RC-1 requirements and providing the response team with the severity basis for containment decisions.

SI-02 Initial Triage Documentation Application

Two components. SI-02 closes Bucket 1 by ensuring the triage record is complete before containment begins. The two-component split reflects a practical distinction: DC-14 is about producing the correct content, and DC-15 is about verifying the record is usable by the team members who will need it next. Both can fail independently. A complete record that is inaccessible to the incoming containment team fails DC-15 even if it passes DC-14.

DC-14 Application

Complete the MRHN triage record with all required initial data elements: detection timestamp, preliminary incident type, severity classification with rationale, escalation status with decision rationale, and assigned responder identification, per RC-1 and RC-2 documentation requirements.

DC-15 Application

Verify triage documentation completeness and confirm the incident record is accessible to all incoming response team members before transitioning to Bucket 2 containment procedures.

SI-01, SI-16, SI-02 Decomposition Complete: 7 components produced

The three remaining Bucket 1 items decomposed into seven components, completing DC-01 through DC-15 for the bucket.

06

Governance Record MRHN-D-005: Bucket 1 Decomposition

DC-01 through DC-15 are now locked as the Bucket 1 portion of MRHN_DecompositionMap_v1. The governance record documents the key decisions, the constraints that traveled from Step 4 and Step 5, and the risks that Module 2 designers must not introduce during instructional engineering.

MRHN-D-005 Bucket 1 Decomposition Decision Record
Decision
DC-01 through DC-15 established as the Bucket 1 component set for MRHN_DecompositionMap_v1. Five Bucket 1 scope items decomposed into 15 components. SI-08 produces four components; SI-06 produces four components. SI-16 produces three components at Analysis level due to contextual evaluation demand. SI-01 and SI-02 produce two components each at Application level.
Calibration Constraints Applied
  • SI-08 (MRHN-D-003): DC-01, DC-03, and DC-04 require evaluative cognitive demand: active hypothesis comparison, ambiguous-indicator practice, and rationale documentation that names ruled-out alternatives. No component from SI-08 can be satisfied by pattern recognition or single-category identification.
  • SI-06 (MRHN-D-003): DC-10 and DC-11 require capability self-assessment before criteria application. DC-13 documentation must name capability factors, not just state the escalation conclusion. Criteria application (DC-12) is Application level but depends on prior Analysis-level work in DC-10 and DC-11 to be valid.
  • SI-16: DC-08 contextual evaluation may override the rubric output. Module 2 must design scenarios where rubric and context point in different directions to develop this capability.
Arc Traceability
All 15 components are traceable to the validated Bucket 1 sub-arc. The "evaluate indicators to distinguish" clause is built by DC-01 through DC-04. The "apply the MRHN severity rubric" clause is built by DC-07 through DC-09. The "evaluate whether Tier 1 capacity is sufficient" clause is built by DC-10 through DC-13. DC-05, DC-06, DC-14, and DC-15 provide the procedural and documentation framework that the arc's "documented severity classification" and "criteria-based rationale" language requires.
Risk Note
  • Risk 1: If Module 2 designers collapse DC-10 and DC-11 into DC-12 (treating escalation as a single criteria-application task), the SI-06 calibration constraint is violated. The capability-assessment and complexity-evaluation components must remain distinct in instructional design.
  • Risk 2: If DC-03 assessments use unambiguous indicator sets (single hypothesis obvious from evidence), the SI-08 calibration constraint is not met. The population-specific gap identified in MRHN-D-003, the absence of practiced pattern recognition, can only be addressed by genuine ambiguity in assessment scenarios.
  • Risk 3: DC-04 and DC-13 documentation components must not be treated as completion checkboxes. Assessment rubrics for both must verify that the rationale names evaluated alternatives and named criteria, not just the conclusion.
Checkpoint Linkage

  • Gate 1.TG.1: MRHN_DecompositionMap_v1 is a required gate artifact. Bucket 1 decomposition is complete; Buckets 2 through 4 completed in Screen 4.
  • Module 3 Outcome Mapping: flagged components (DC-01, DC-03, DC-04, DC-08, DC-10, DC-11, DC-13) require assessment alignment verification: each must have a corresponding assessment item that targets the evaluative demand, not just the procedural outcome.

Expert Think-Aloud

01

The easiest mistake in decomposition is producing components that look right at the content level but ignore what the calibration says about cognitive demand. A list of correct identification behaviors is not the same as a decomposition that builds evaluation capability. The taxonomy level tells me what type of practice each component must require: that is a design specification, not a label.

02

DC-03 is the hardest component in Bucket 1 to design instruction for, which is precisely why it must exist as a separate component. If I fold ambiguous-indicator practice into DC-01 or DC-02, it will get under-weighted in Module 2. Decomposition creates explicit instructional obligations. A component that exists cannot be skipped. A capability folded into another component often is.

03

The "naive two-component" decomposition of SI-06 I described is what you get when you decompose from a generic scope item label rather than from the calibrated item. The label says "escalation decision." The calibrated item says "evaluate team capability against incident demand." Those are not the same instructional problem, and they produce fundamentally different component sets.

04

SI-16 is Analysis level, but it gets three components rather than four because the "apply the rubric" and "evaluate context" steps do not have a hypothesis-evaluation equivalent. The contextual evaluation (DC-08) is the Analysis-level demand for this item. The third component (DC-09) is documentation of the result. The item does not require a fourth component because the taxonomy placement is anchored to one specific analytical act, not to a full analytical process across multiple steps.

05

The calibration flag column in the reference table is not decoration. It is an instruction to the Module 2 designer: this component requires a specific instructional treatment that a generic activity cannot provide. When I hand MRHN_DecompositionMap_v1 to the Module 2 team, the flags tell them where they cannot use a standard activity template; they must design something that creates genuine evaluative demand.

06

The risk notes in MRHN-D-005 are written to anticipate handoff failures. I have seen all three of these failure modes in actual decomposition projects: collapsed Analysis components, sanitized assessment scenarios, and treated documentation rubrics as completion checkboxes. Naming them now in the governance record means they are available for Gate 1 review and Module 3 stress-testing, not discovered in a learner assessment that fails to produce the capability the course promised.

In a Different Context

Picture a nonprofit teaching financial literacy to adult learners. During scoping, the team notes that many learners distrust banks after past harm. That note is not background color. It becomes a constraint: every budgeting component must build trust and ease fear, not just teach a procedure. The same principle holds here. A calibration note governs what each component must require.

Principle Extraction

What is the governing rule for deciding how many components a scope item should produce?

Saved

Reference answer

A scope item should produce as many components as the performance requires, no more and no fewer. Each component must pass two tests: it can be taught as a discrete unit, and it can be assessed independently. Components that fail either test should be merged, split, or removed.
