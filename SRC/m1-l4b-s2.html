Lesson 1.4b

Unit Optimization: Testing Whether Each DC Component Is the Right Size

Phase 2 -- Step 7
Scenario: Meridian Regional Health Network. MRHN_DecompositionMap_v1 (DC-01 through DC-48) is the input. An expert applies the two-question optimization test to selected components, splits one component that bundles two distinct assessment demands, merges one component that cannot be assessed independently, and confirms one component that is correctly granular. Produces governance record MRHN-D-007.
01

The Optimization Criteria -- Two Questions Every Component Must Answer

Unit optimization is not a judgment about whether a component is important. It is a structural check: is this component the right unit for independent instruction and assessment? A component that covers too much cannot be assessed cleanly -- the assessment item would be measuring two or three distinct things at once. A component that covers too little has no instructional value on its own -- it is a sub-behavior that only makes sense embedded inside a larger skill.

Every DC component in MRHN_DecompositionMap_v1 must pass both questions before the map advances to Step 8. A fail on either question triggers a split (if too large) or a merge (if too small). The test does not require consulting the calibration annotations -- but any optimization decision that affects a component with a calibration constraint must preserve that constraint in the resulting component or components.

Question 1 Can this be taught as one instructional unit?
Teachability: a single instructional unit addresses one skill or one cluster of tightly coupled sub-behaviors that have no meaning in isolation. If a component requires a learner to perform two distinct cognitive operations that could each be the subject of a standalone learning activity, the component is too large. The signal: a designer building instruction for this component would have to choose between two different activity types because the component spans two different kinds of performance.
Question 2 Can this be assessed by one well-formed item?
Assessability: a well-formed assessment item tests one competency claim. If writing an item for this component produces a compound question ("can the learner do X and Y"), the component is too large and must be split. Conversely, if the only plausible item would be testing a lookup behavior ("can the learner find the document that lists X"), the component is too small -- that behavior is embedded in whatever skill uses the document, not a learnable competency in its own right.

I will work through three representative cases from MRHN_DecompositionMap_v1: one component that fails by being too large, one that fails by being too small, and one that passes both questions and should not be changed. The same logic applies to the remaining 45 components not shown here -- Step 7 requires running both questions against every component in the map.

02

Case 1: DC-18 Must Be Split -- Two Distinct Assessment Demands

DC-18 comes from Bucket 2, parent scope item SI-09 (Evaluate MRHN clinical system interdependencies for network segmentation risk). SI-09 carries a domain authenticity annotation from Step 4: network segmentation at MRHN must account for EHR, imaging, and billing system relationships -- the Analysis-level demand comes from having to evaluate those relationships before applying any segmentation procedure, not just from knowing the procedure itself. I need to check whether DC-18 as written preserves that analytical demand in a single assessable unit.

DC-18 Parent: SI-09 • Bucket 2 • Analysis level Split Required
Component Description (v1)

"Evaluate clinical system interdependency relationships at active MRHN facilities to identify network segmentation risks to patient care systems, and document the containment decision rationale with reference to identified interdependencies and PC-3 authority boundaries."

Q1: Can this be taught as one instructional unit?

Fail. The component spans two different activity types: a clinical system analysis activity (evaluate interdependency relationships) and a documentation activity (document decision rationale). A designer building instruction for DC-18 would have to choose between a scenario-based analysis task and a structured documentation exercise -- because the component requires both.

Q2: Can this be assessed by one well-formed item?

Fail. A single assessment item would have to ask "can the learner both evaluate interdependency risks AND produce a documented rationale?" -- a compound question measuring two competency claims. A learner could correctly analyze the interdependencies and then produce a non-compliant rationale document, or vice versa. These are separately assessable.

Q1: Fails -- two activity types Q2: Fails -- compound assessment

DC-18 bundles an analytical competency (evaluate interdependency relationships to identify segmentation risk) with a documentation competency (produce a rationale record). These are different cognitive operations: the first is Analysis-level evaluation of a complex system relationship; the second is Application-level structured documentation. Combining them means the calibration annotation from Step 4 -- which governs the Analysis-level evaluation -- is diluted by a lower-level documentation task. The split is required both for instructional coherence and to preserve the SI-09 calibration constraint in its correct component.

Resolution: DC-18 Split into DC-18a and DC-18b
DC-18a Parent: SI-09 • Analysis level • Calibration annotation applies

Evaluate clinical system interdependency relationships (EHR, imaging, and billing systems) at the affected MRHN facility to identify patient care system exposure risks before applying network segmentation procedures.

DC-18b Parent: SI-09 • Application level • PC-3 authority constraint applies

Document the network segmentation containment decision with reference to identified clinical system interdependencies, the rationale for scope of isolation applied, and the PC-3 authority boundary governing the decision.

After the split: DC-18a carries the Analysis-level calibration constraint from SI-09 (the clinical system interdependency evaluation is the authenticity-critical skill). DC-18b is Application level -- producing a compliant documentation record -- and references PC-3 authority explicitly. The split adds one component to the map (48 becomes 49 at this point); the merge in Case 2 restores the count to 48.

03

Case 2: DC-07 Must Be Merged -- Not Independently Assessable

DC-07 comes from Bucket 1, parent scope item SI-01 (Incident classification using the MRHN severity rubric). SI-01 sits at Application level in the calibrated scope map -- the skill is applying the rubric to classify a potential incident, not describing what the rubric contains. I look at DC-07 and DC-08 together because they were decomposed from the same scope item and the test applies to whether each component is independently assessable.

DC-07 and DC-08 Parent: SI-01 • Bucket 1 • Application level Merge Required
DC-07 (v1 description)

"Locate the MRHN incident severity rubric within the IR-2024 protocol documentation and identify the four severity classification criteria."

DC-08 (v1 description)

"Apply the MRHN severity rubric criteria to a potential incident event to assign a documented severity level."

Q1 applied to DC-07: Can this be taught as one instructional unit?

Borderline. "Locate a document and identify its criteria" is a single teachable behavior. But the question is whether it has instructional value as a standalone unit.

Q2 applied to DC-07: Can this be assessed by one well-formed item?

Fail. The only plausible item would test lookup behavior: "open IR-2024 and locate the rubric." That is not a learnable competency -- it is an embedded prerequisite behavior within DC-08. A learner who cannot locate the rubric cannot perform DC-08, so the inability will surface there. Assessing DC-07 independently adds nothing to the architecture.

Q2: Fails -- lookup behavior only, not independently assessable

The problem with DC-07 is not that locating the rubric is unimportant -- it is that locating a document is not a competency that belongs in the map as a standalone unit. At Application level, the SI-01 skill is using the rubric to classify incidents, not finding the rubric. "Locate and identify" is the prerequisite behavior embedded within "apply." If DC-07 traveled to Module 2 as a standalone component, a designer would be asked to build an activity for it -- and there is no meaningful activity to build beyond a lookup exercise that tests nothing relevant to incident classification competence. DC-07 must be merged into DC-08, which already captures the application-level skill that the scoping of SI-01 requires.

Resolution: DC-07 Retired -- Merged into DC-08
DC-07 (retired) Designation retired

"Locate the MRHN incident severity rubric within the IR-2024 protocol documentation and identify the four severity classification criteria."

merged into
DC-07 Parent: SI-01 • Application level • (formerly DC-08, renumbered)

Apply the MRHN incident severity rubric criteria (IR-2024) to a potential incident event to assign and document a severity level, distinguishing security incidents from non-incident system events using the rubric's four classification criteria.

After the merge: the revised DC-07 incorporates "using the rubric's four classification criteria" as embedded context -- the locating behavior is implied by application of the criteria. The "distinguishing security incidents from non-incident system events" language connects this component to the SI-01 calibration: the rubric is the tool, but the distinction-making is the skill. Net component count: DC-07 retired and DC-08 renumbered to DC-07, returning the map to 48 components after the DC-18 split added one.

04

Case 3: DC-35 Is Correctly Granular -- Do Not Change It

DC-35 comes from Bucket 3, parent scope item SI-10 (HIPAA 72-hour breach notification clock management). SI-10 is one of the highest-stakes components in the entire map: it carries the domain authenticity annotation from Step 4 identifying the "reasonably should have known" discovery trigger under 45 CFR 164.404, and the Bucket 3 sub-arc was revised in Step 5 specifically because of this component. I approach DC-35 with heightened attention -- a component that looks narrow may still be covering the essential analytical skill that makes the HIPAA compliance training meaningful.

DC-35 Parent: SI-10 • Bucket 3 • Analysis level • Calibration annotation: MRHN-D-003 Confirmed Correct
Component Description (v1)

"Evaluate the facts of a potential security incident to determine whether the 'reasonably should have known' discovery trigger under 45 CFR 164.404 has been met, and identify the point at which the HIPAA 72-hour breach notification clock begins."

Q1: Can this be taught as one instructional unit?

Pass. The component asks learners to do one thing: evaluate incident facts against the legal standard for the discovery trigger. "Identify the clock start point" is the direct consequence of that evaluation -- it is not a separate cognitive operation but the output of the same analytical judgment. This is one instructional unit.

Q2: Can this be assessed by one well-formed item?

Pass. A single scenario-based item can present incident facts and ask: "At what point should MRHN have determined they reasonably knew of the potential breach, and when does the 72-hour clock begin?" This is one competency claim -- the trigger evaluation -- with one expected reasoning output.

Q1: Pass -- single analytical unit Q2: Pass -- one well-formed item possible

The optimization instinct here might be to split DC-35: "evaluate trigger conditions" and "identify clock start point" could look like two components. That split would be wrong. Identifying the clock start point is not a separable skill -- it is the outcome of the trigger evaluation, not an independent competency. A learner who correctly evaluates the trigger has by definition identified the start point. Splitting here would create a DC component ("identify the point at which the clock begins") that could only be assessed by restating DC-35's scenario item -- making it a redundant component, not an independent one.

Why the impulse to split DC-35 is wrong

The calibration annotation on SI-10 names the "reasonably should have known" trigger as the authenticity-critical skill -- the one that generic HIPAA training gets wrong. That critical skill is exactly what DC-35 covers, and only what it covers. Any split would either dilute the trigger evaluation into a lower-stakes component or create a downstream component with no independent instructional value. DC-35 is correctly granular because it was decomposed from an authenticity-annotated item by a practitioner who read the annotation. The optimization check confirms that -- it does not override it.

Optimization Summary: What Changed Across All 48 Components

The three cases above are representative, not exhaustive. Every component in MRHN_DecompositionMap_v1 was evaluated against both optimization questions. Two components required action. The remaining 46 were confirmed without change. The net component count remains 48: one split (+1) is offset by one merge (-1), but two DC designations changed (DC-07 revised and renumbered; DC-18 retired in favor of DC-18a and DC-18b).

Component (v1) Parent SI Optimization Action Result (v2)
DC-07 (v1) SI-01 • Bucket 1 Merged DC-07 retired -- lookup behavior embedded into application component DC-07 revised (formerly DC-08); DC-08 designation retired
DC-18 (v1) SI-09 • Bucket 2 Split Two distinct assessment demands separated: Analysis-level evaluation and Application-level documentation DC-18a (evaluation, Analysis) + DC-18b (documentation, Application); DC-18 designation retired
All other DCs All buckets Confirmed Both optimization questions passed without action Unchanged from v1 (46 components)

The practical significance of these two changes: the DC-18 split ensures the Analysis-level calibration constraint on SI-09 is isolated in DC-18a, where a Module 2 designer will build the clinical interdependency analysis activity -- not buried inside a component that also asks for documentation. The DC-07 merge removes a component that would have generated a low-value lookup activity in Module 2, replacing it with a single application-level component that correctly targets SI-01's competency demand.

06

Governance Record MRHN-D-007: Unit Optimization

The optimization decisions are formalized in MRHN-D-007. This record documents what changed, why the optimization criteria required the change, and what constraints must be preserved in the revised components. It travels with the map into Step 8 and then into Module 2 as the decision record for any question about why DC-18 has two sub-designations or why there is no standalone DC-08.

MRHN-D-007 Unit Optimization Record -- Phase 2 Step 7
Decision
Applied two-question unit optimization criteria to all 48 components in MRHN_DecompositionMap_v1. Two optimization actions taken. DC-18 (SI-09, Bucket 2) split into DC-18a and DC-18b: component bundled an Analysis-level clinical system evaluation with an Application-level documentation task, failing both optimization questions. DC-07 (SI-01, Bucket 1) merged into the revised DC-07 (formerly DC-08): original DC-07 described only a document lookup behavior with no independent assessment value. Net count: 48 components (unchanged). DC-08 and DC-18 designations retired.
Constraint Reference
SI-09 domain authenticity annotation (MRHN-D-003): clinical system interdependency evaluation is the Analysis-level skill; this constraint is now isolated in DC-18a. PC-3 authority boundary (system isolation): referenced explicitly in DC-18b documentation component. SI-01 Application level (MRHN_CalibratedScopeMap_v1): rubric application is the competency claim; lookup behavior is not independently assessable.
Rationale
DC-18 required a split because bundling an Analysis-level evaluation with an Application-level documentation task in a single component dilutes the calibration annotation on SI-09 -- the analytical demand (evaluate interdependencies to identify segmentation risk) would be partially obscured by the documentation demand in a Module 2 design brief. The split surfaces the Analysis-level component cleanly. DC-07 required a merge because "locate a document and identify its criteria" is an embedded prerequisite behavior, not a learnable competency. An assessment item for DC-07 would only test whether a learner can open a document -- not whether they can use it. The DC-08 (now DC-07) application component renders DC-07 (v1) redundant.
Risk Note
Risk 1: If a Module 2 designer combines DC-18a and DC-18b back into a single activity, the Analysis-level calibration constraint on SI-09 will be lost in a mixed-level task. DC-18a and DC-18b must be treated as independent instructional units in Module 2 design briefs.

Risk 2: The DC-08 designation no longer exists. Any reference to DC-08 in existing documentation (notes, drafts, or working files from Lesson 1.4a) refers to the component now designated DC-07 (revised). This must be noted in the Scope Trace Audit conducted in Step 8.
Checkpoint Linkage
  • Step 8 Scope Trace Audit: the audit must use the revised component structure (DC-18a, DC-18b, revised DC-07; DC-08 and DC-18 designations retired). Both DC-18a and DC-18b must trace to SI-09. The revised DC-07 must trace to SI-01.
  • Module 2 Design Briefs: DC-18a must be assigned Analysis-level activity design. DC-18b is Application-level. A design brief that assigns both to the same activity type misreads the optimization.
  • Gate 1.TG.1: MRHN-D-007 is a required gate record. Optimization decisions must be on file before the gate review.

Expert Think-Aloud

01

I apply both questions to every component, not just to the ones I suspect have a problem. The map has 48 components; I have seen experienced designers produce well-formed decompositions where one component in a set of four siblings quietly bundles two skills because both skills were on the same training slide in a predecessor course. The problem is invisible until you ask whether the component can be assessed by a single item.

02

The signal that DC-18 needed a split was the phrase "and document." That conjunction is almost always a warning sign in a component description. "Evaluate X and document Y" is two competencies unless documentation is so tightly coupled to the evaluation that it cannot be assessed separately. Here it can be: a learner who correctly evaluates the interdependency risk might still produce a non-compliant documentation record. Those are different failure modes, which means they need different components.

03

DC-07 is the kind of component that appears in first-draft decompositions produced by designers who are still thinking in content terms ("learners need to know where the rubric is") rather than performance terms ("learners need to apply the rubric"). The test question cuts through it: can you write one non-trivial item for this? If the only item is "find the rubric in IR-2024," the component does not belong in the map. The rubric-finding behavior will be established in the activity for DC-07 (revised) -- it does not need its own component.

04

I spend extra time on DC-35 because the calibration annotation creates pressure to split. The "reasonably should have known" trigger seems like it could be separated from "identify when the clock starts." The reason I do not split is that identifying when the clock starts is not an independent cognitive operation -- it is the immediate output of the trigger evaluation. There is no assessable gap between "you have evaluated the trigger" and "you know when the clock starts." Splitting here produces a redundant item, not an independent competency.

05

The two changes are not arbitrary revisions to the decomposition -- they are corrections to structural problems that would have cost Module 2 designers time and produced weaker instruction. If DC-18 had reached Module 2 unchanged, a designer building that component would have to make an implicit split decision during activity design with no documented rationale. Doing it here, with the calibration record in front of me, produces a better decision than one made under Module 2 time pressure.

06

MRHN-D-007 is a short record because the optimization changes were limited. In a messier decomposition -- one that produced a first draft with five or six over-bundled components -- this record would be much longer. The length of the optimization record is a rough indicator of decomposition quality from Step 6. A short MRHN-D-007 means the Step 6 decomposition was disciplined. A long one means the decomposition needed significant repair before it could be audited.
