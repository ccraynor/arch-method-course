Lesson 2

Output Style Gate and Snapshot Protocol

Step 1.5
Scenario: You have extracted and ranked the Meridian constraints. Now watch an expert work before any architecture begins. The expert establishes five MRHN documentation conventions. All artifacts, identifiers, decisions, and version records will follow them through Structural Freeze.

Why would an expert spend time naming files and numbering identifiers before designing a single thing? The order here is deliberate, not busywork.

01

Establish the Artifact Naming Convention

The first thing I establish before producing any document is the naming convention. Every MRHN artifact, from the Bucket Map produced in this lesson through the Structural Freeze Declaration Package in Module 4, must follow the same pattern from the first version forward.

Convention Artifact Naming
MRHN_[ArtifactName]_v[Version]_[Date] Example: MRHN_BucketMap_v1_2026-01-15

Four required segments, separated by underscores. No spaces. Date format: YYYY-MM-DD. Version number is an integer starting at 1. ArtifactName uses CamelCase with no spaces or hyphens.

Without this convention: A project running across two IDs produces BucketMap-Final, BucketMap-REVISED, and BucketMap-CarrieVersion. No one knows which is current at the SME review.

I establish this now, before the first artifact is produced, because retroactive renaming creates confusion about which version is current. Once a client or governance reviewer has seen a file named MRHN_BucketMap_v1_2026-01-15, that filename is a stable reference point. If I produce a v2, it becomes MRHN_BucketMap_v2_2026-01-28. Both files exist. The history is preserved in the filenames without opening either document.

The complete set of MRHN artifacts that will follow this convention across the credential:

Artifact Filename Pattern
Bucket Map and Scope Organization Package MRHN_BucketMap_v1_[Date]
Sequenced Unit Architecture Package MRHN_UnitArchitecture_v1_[Date]
Audit and Governance Package MRHN_AuditGovernance_v1_[Date]
Structural Freeze Declaration Package MRHN_StructuralFreeze_v1_[Date]
02

Establish the SI-XX Scope Item Identifier System

Every distinct item within the declared scope receives a sequential scope item identifier: SI-01, SI-02, and so on up to SI-16. These identifiers are assigned when the Bucket Map is built in Screen 5. I establish the convention now so that assignment is consistent from the first entry.

Convention Scope Item Identifiers
SI-[##] Range: SI-01 through SI-16 for MRHN

Sequential. Assigned in the order scope items appear in the Bucket Map. Never reused. If a scope item is removed, its identifier is retired: the sequence does not collapse to fill the gap.

Notice this: a removed identifier is retired for good, never reused to close the gap.

Without this convention: A Module 3 audit finding that references the scope item about role-bounded authority cannot be traced back to the source constraint without reading every scope item description.

The value of SI identifiers becomes clear in governance documentation. Instead of writing "the requirement that learners execute Tier 1 and Tier 2 IR-2024 protocol decisions independently within their role authority" in every decision record, I write SI-04, or whichever number that item receives in the Bucket Map. The reference is unambiguous, traceable, and auditable.

A sample of what SI assignment looks like in the Bucket Map:

Identifier Scope Item Constraint Source
SI-01 IR-2024 Tier 1 protocol execution: independent, no escalation PC-1, PC-2, RC-2
SI-02 IR-2024 Tier 2 protocol execution: within role authority PC-2, PC-3
SI-03 System isolation decisions: role-bounded, no supervisor required PC-3
SI-04 Real-time SIEM event log documentation PC-4
SI-05 Clinical leadership communication using Meridian approved templates PC-5
03

Establish the DC-XX Decomposition Component Identifier System

When scope items are decomposed into learning components in Module 2, each component receives a sequential decomposition component identifier: DC-01, DC-02, and so on. For the MRHN project, the decomposition will generate up to 48 components across all scope items.

Convention Decomposition Component Identifiers
DC-[##] Range: DC-01 through DC-48 for MRHN

Sequential. Assigned during decomposition in Module 2, grouped by parent SI identifier. DC identifiers follow the same retirement rule as SI identifiers: removed components are retired, not renumbered.

Without this convention: An optimization decision to split a component in Week 3 cannot be referenced in the Module 4 governance record without describing the component from scratch.

I establish DC-XX now, before decomposition, so that Module 2 work begins with a known identifier system in place. The relationship between SI and DC identifiers is explicit: DC-01 through DC-06 might belong to SI-01; DC-07 through DC-11 to SI-02. That relationship is recorded in the Sequenced Unit Architecture Package.

DC identifiers become critical in Module 3 stress-testing. When the Outcome Mapping check surfaces a gap, the finding is recorded against a specific DC identifier. Without component-level identifiers, stress-test findings are imprecise: "there is a gap in the isolation content" is not actionable. "DC-07 does not address role-bounded authority for Tier 2 decisions" is.

04

Establish the Five-Field Governance Documentation Rule

Every design decision recorded in the MRHN decision log must contain five fields. Not four. Not "most of them." All five, for every decision, without exception. A decision record with missing fields is not a governance document: it is a note.

Convention Governance Decision Record
  1. Decision Statement: what was decided, written as a precise present-tense declaration
  2. Constraint Reference: which SI-XX identifier or named binding constraint governs this decision
  3. Rationale: why this decision follows from the constraint; not a restatement of the decision
  4. Risk Note: what could go wrong if this decision is incorrect or needs to change
  5. Governance Checkpoint Linkage: which checkpoint will verify or revisit this decision

Without this convention: The SME signs off on a decomposition decision without knowing that the 4-hour access constraint makes one of the components unrealistic to develop.

Here is the first completed governance record for the Meridian project: the decision about IR-2024 version alignment:

Decision Record: MRHN-D-001
1 Decision Statement
The MRHN course will align to IR-2024 as adopted in January 2024. The course will not incorporate revisions to IR-2024 that occur after project start without a documented scope change.
2 Constraint Reference
RC-2 (IR-2024 as adopted); PC-1 (IR-2024 protocol alignment); LC-1 (4 hours total SME access)
3 Rationale
The OCR remediation timeline (RC-3) and the SME access cap (LC-1) mean scope cannot flex after the first SME meeting. Aligning to IR-2024 as a fixed document prevents scope drift that cannot be absorbed within project constraints. The client's brief specifies IR-2024 by name and adoption date; this decision honors that specification.
4 Risk Note
If Meridian updates IR-2024 during the project window, the delivered course will not reflect the change. This risk is accepted given the timeline and SME constraints. Post-launch protocol updates require a documented revision under the version control protocol (v2 minimum).
5 Checkpoint Linkage
Gate 1.TG.1 (Transition Gate, Module 1) will verify that all performance items trace to IR-2024 as adopted. Module 3 Outcome Mapping will confirm no performance item references a protocol version not covered by the course content.
05

Establish the Version Control Rule

Architecture revisions occur only after documented governance review. This is the version control rule, and it has one purpose: to ensure that every change to the architecture is traceable, not just the final state.

Version Control Protocol

  1. A revision is triggered by a documented governance finding: a stress-test gap, a checkpoint failure, a client-requested change, or a scope adjustment
  2. Before revising, record what is changing, which constraint reference or checkpoint triggered the review, the risk of not changing, and the risk of changing
  3. Create a new file with the next version number and a new date: MRHN_BucketMap_v2_2026-02-03
  4. Do not edit v1 in place: v1 remains as the permanent record of the original design decision
  5. Update the governance decision log to reference the new version and record the trigger for the revision

Without this convention: The content developer cannot tell whether the architecture document in their email is the pre-SME version or the post-review version.

The version control rule exists because architecture documents get edited. An overwritten file has no audit trail. When a governance reviewer asks "what did the architecture say before this change, and why was it changed?", the answer must be findable without reconstructing history from memory.

These five conventions, naming, SI identifiers, DC identifiers, the governance documentation rule, and version control, are the Output Style Gate. Once established, they govern every artifact produced between now and Structural Freeze. The Bucket Map built in Screen 5 is the first artifact to follow them.

Expert Think-Aloud

01

Naming conventions feel administrative until you have five versions of the same document with names like "Architecture Final v3 REAL (2).docx" and a client asking which one is current. The convention is not overhead. It is the minimum condition for a document to be a governance artifact rather than a file.

02

SI identifiers cannot be assigned before scope is declared. This is why the sequence matters: Step 0 declares scope, Step 1 extracts constraints, Step 1.5 establishes the identifier system, and then Screen 5 assigns SI identifiers to the declared scope items. Doing Step 1.5 before Step 0 would produce identifiers with nothing to reference.

03

Establishing DC-XX now assigns no DC identifiers. It reserves the naming system so that when Module 2 produces components, there is no question about how to label them. The alternative, deciding on an identifier format during decomposition, introduces inconsistency at the worst possible moment: when the volume of decisions is highest.

04

The Risk Note is the field most commonly skipped. It requires the designer to identify what could go wrong with their own decision before a reviewer does. That discomfort is the point. A designer who cannot articulate the risk of their own decision has not thought the decision through. The Risk Note is where expert judgment becomes visible.

05

"Final" is not a version number. The version control rule exists because the instinct under deadline pressure is to edit the existing file. Once you overwrite v1, the original decision is gone. The five-step version control protocol is not slower than editing in place: the governance record write takes four minutes. The audit conversation when someone asks why the architecture changed takes four hours.

In a Different Context

Picture a community college first-year experience course for underprepared students. Before any module is written, you set naming rules, unit identifiers, and a shared decision log. When an advisor later asks why the study-skills unit was cut, the record answers in seconds. The convention protects the design long after the original reasons fade from memory.
