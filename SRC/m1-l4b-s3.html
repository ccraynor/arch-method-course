Lesson 1.4b

Scope Trace Audit: Confirming Every Component Traces to a Parent Scope Item

Phase 2 -- Step 8
Scenario: Meridian Regional Health Network. The optimized MRHN decomposition map (48 components, MRHN-D-007 on record) is the input. An expert runs the three-pass Scope Trace Audit -- forward trace, backward trace, orphan check -- discovers one parent misattribution during the backward trace, resolves it, and confirms the audit passes with zero orphan components. Produces MRHN_DecompositionMap_v2 and governance record MRHN-D-008.
01

The Three-Pass Audit -- What Each Pass Catches and Why One Pass Is Not Enough

The Scope Trace Audit answers one question: does the decomposition map stay within the scope boundary? A component that cannot be traced to a parent SI item in the scope registry is content outside the boundary -- something that was added during decomposition without authorization from Step 3 scoping decisions. Equally, a scope item with no DC children means a piece of authorized scope was silently dropped during decomposition. Either condition represents a structural failure in the map before it reaches Module 2.

The audit runs as three sequential passes. Running only one or two passes is insufficient: each pass catches a different class of failure, and the orphan check requires both previous passes to be complete before it can run.

Pass 1 Forward Trace
SI registry to DC components. Start from each of the 16 SI items. Count how many DC components list that SI item as their parent. Every SI item must have at least one DC child -- a count of zero means a scope item was dropped during decomposition. Note unusually high counts for verification in Pass 2. Pass 1 does not verify that component descriptions match their listed parents -- that is Pass 2's job.
Pass 2 Backward Trace
DC components to SI registry. For each of the 48 DC components, verify that (a) a parent SI item is listed, (b) that SI item exists in the scope registry, and (c) the component description is consistent with what that SI item covers. A misattribution -- a component whose description does not match its listed parent -- is a labeling error that Pass 1 cannot catch. Pass 2 is the only pass that reads both the component description and the parent SI item name together.
Pass 3 Orphan Check
Cross-reference. After any corrections from Pass 2 are applied, run the cross-reference: confirm that every SI item in the registry appears at least once as a parent in the corrected backward trace, and that every DC component in the corrected map appears as a child in the forward trace. A correction in Pass 2 that moves a component to a different parent SI can leave the original parent unrepresented -- Pass 3 is what catches that knock-on effect.
02

Pass 1 -- Forward Trace: All 16 SI Items Have at Least One DC Child

I run the forward trace by reading down the SI registry and counting how many DC components list each SI item as their parent. The pass criterion is simple: every SI item must return a count of at least one. I note the count for each SI item and flag any that look unusually high or low for follow-up in Pass 2. The forward trace does not verify description alignment -- that happens in Pass 2.

03

Pass 2 -- Backward Trace: DC-45 Misattribution Found

Pass 2 reads every component description and checks it against its listed parent SI item. For the 45 components not flagged in Pass 1, the check is confirmatory and moves quickly -- the description matches the parent. I run the full verification but spend additional time on the SI-15 components because of the count flag. When I reach DC-45, the mismatch is immediate.

The mismatch is unambiguous: DC-45 describes a lessons learned documentation task explicitly naming the operational/governed record distinction -- which is the defining characteristic of SI-13, not SI-15. SI-15 covers recovery verification: checking whether systems have met restoration criteria before declaring the incident closed. There is no overlap between the two scope items, so the misattribution cannot be resolved by adjusting the component description -- the parent label must be corrected. Before making the correction, I need to verify that SI-15 will still be represented in the map after DC-45 is moved.

04

Investigation: Confirming SI-15 Remains Represented After Correction

Before I correct DC-45, I check what happens to SI-15 if DC-45 is removed from its child list. Pass 1 showed SI-15 had four DC children. If DC-45 is moved to SI-13, SI-15 will have three remaining children. I verify all three by reading their descriptions.

SI-15 Remaining Children After DC-45 Correction

DC-43 (parent: SI-15): "Apply MRHN system restoration criteria to evaluate whether affected systems meet functional and security requirements before declaring them ready for reintegration." -- Recovery verification. Correctly attributed to SI-15.

DC-44 (parent: SI-15): "Document the restoration verification outcome for each affected system, recording which criteria were tested, the test result, and the approving authority under PC-3." -- Recovery documentation. Correctly attributed to SI-15.

DC-46 (parent: SI-15): "Evaluate whether all MRHN facility systems have met restoration criteria before authorizing the transition from incident response mode to normal operations." -- Multi-facility restoration authorization. Correctly attributed to SI-15.

SI-15 retains three correctly attributed DC children after the DC-45 correction. No orphan scope item will result from the correction.

With SI-15 confirmed to remain represented, I apply the correction and verify the outcome for SI-13.

Correction Applied: DC-45 Parent Attribute
DC-45 parent: SI-15 → DC-45 parent: SI-13

After correction, SI-13 now has three DC children: DC-45 (corrected), DC-47, and DC-48. All three descriptions align with SI-13 (post-incident lessons learned documentation, operational/governed record distinction, Evaluation-level judgment).

Why misattribution happens: DC-45 was the last component added to the Bucket 4 decomposition. The working document had SI-15 as the active parent field when DC-43, DC-44, and DC-46 were entered. DC-45 was added in a subsequent session, inheriting the SI-15 parent from the prior entry without being updated when the lessons learned content was drafted under a different scope item. This is the most common misattribution pattern in decomposition maps built across multiple sessions: the parent field carries over from the previous entry. The backward trace is the only systematic check that catches it.

05

With the DC-45 correction applied, I run the cross-reference orphan check. The check runs against the corrected map, not the original. Three conditions must all be true for the audit to pass.

All 16 scope items represented. All 48 components traced to valid parent SI items. Zero orphan components. One misattribution corrected (DC-45: parent updated from SI-15 to SI-13). Map is cleared for advancement as MRHN_DecompositionMap_v2.

It is the artifact that Module 2 will use to assign DC components to instruction sequences. No further changes are permitted to the structural map without a new governance record.

Constraint Reference
LC-1 (scope boundary: only content traceable to SI-01 through SI-16 may appear in the decomposition map); MRHN_ScopeRegistry_v1 (authoritative SI item list and descriptions, established in Step 3); MRHN-D-007 (Step 7 optimization record, structural changes incorporated into v2 before audit); SI-13 Evaluation-level assignment (MRHN_CalibratedScopeMap_v1): DC-45 must reflect that level in Module 2 design brief.
Rationale
The DC-45 misattribution was a labeling error introduced during the final session of Step 6 decomposition: the parent field carried the SI-15 value from the previous entry (DC-44) without being updated when DC-45 was drafted under SI-13. The backward trace is the only systematic process that reads component descriptions against their listed parents -- the error was undetectable in the forward trace because SI-15 had the correct count before the DC-45 correction. This confirms the necessity of running Pass 2 even when Pass 1 produces no failures.
Risk Note
Risk 1: Any Module 2 design brief drafted from MRHN_DecompositionMap_v1 (pre-correction) will assign DC-45 to SI-15 instruction, producing recovery verification instruction for a lessons learned component. All Module 2 work must reference v2 as the structural input.

Risk 2: The optimization designations from MRHN-D-007 (DC-18a, DC-18b, revised DC-07; DC-08 and DC-18 designations retired) are structural. Any Module 2 reference to the retired designations (DC-08, DC-18) must be flagged and corrected to the v2 identifiers.
Checkpoint Linkage
  • Gate 1.TG.1 (Module 1 Threshold Check): MRHN_DecompositionMap_v2 and MRHN-D-008 are required gate artifacts. Gate review will verify that the audit was conducted on the post-optimization map (not v1) and that all corrections are documented.
  • Module 2 design briefs: each brief references the DC component from v2 as the structural unit. DC-45 brief must reflect SI-13 (lessons learned, Evaluation level), not SI-15.
  • Module 3 Outcome Mapping: the trace record established here is what makes Outcome Mapping tractable in Module 3 -- assessments can be traced backward through DC components to SI items to scope decisions because this audit confirmed the trace is clean.

Expert Think-Aloud

01

The reason I cannot run just one pass is that forward and backward traces catch different failure modes. A forward trace that finds every SI item has at least one child cannot tell me whether any of those children have descriptions that match the parent. A backward trace alone cannot tell me whether a scope item was dropped entirely -- if it was, there are no components to trace backward from. Both passes are necessary; neither is sufficient.

02

I flag SI-15 for a count check not because four is too many -- some scope items legitimately have more children -- but because the Bucket 4 decomposition from Step 6 produced three SI items with roughly equal component counts. A count of four when the siblings have three is worth verifying. This is pattern recognition, not a rule: it tells me where to spend extra attention in Pass 2. The actual check happens in Pass 2.

03

The DC-45 mismatch becomes obvious the moment I read the description next to the parent label. "Lessons learned, distinguishing operational notes from the formal governed record" is the defining language of SI-13 -- it came from the Bucket 4 sub-arc revision in Step 5. SI-15 is recovery verification. There is no interpretation that makes these the same scope item. This is not an edge case requiring judgment -- it is a labeling error that the description makes immediately visible.

04

The step of checking SI-15's remaining children before correcting DC-45 is not optional. If SI-15 only had one child (DC-45), moving DC-45 to SI-13 would leave SI-15 with zero children -- an unrepresented scope item, which is a more serious finding than a misattribution. I always verify the donor SI item before making the correction. In this case SI-15 has three other solid children, so the correction is safe.

05

Pass 3 is not redundant with Passes 1 and 2. It runs against the corrected map, not the original. Any correction made in Pass 2 changes the trace structure, and Pass 3 is what verifies that the correction did not introduce a new problem. In this case it did not -- but there are decompositions where a correction cascade is possible: moving a component to a different parent can reveal that the receiving parent is now over-populated, prompting a second round of Pass 2 work.

06

MRHN_DecompositionMap_v2 is the last artifact produced before the module-level gate. Its integrity matters more than any other artifact produced in Phase 2 -- it is the structural input to everything that follows. If v2 has an orphan component or an unrepresented scope item, that error propagates through Module 2 instruction design, Module 3 stress-testing, and eventually into the assessment architecture. A clean audit here is not bureaucratic -- it is the condition that makes Module 2 tractable.
